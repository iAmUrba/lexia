# RFC-011 — API Contracts (Platform Interfaces)

**Metadatos de Ingeniería Organizacional:**
- **Autor Principal:** Staff Backend Engineer
- **Revisores:** CTO, Frontend Lead, Integration Lead
- **Aprobadores:** Chief Architect (CTO)
- **Estado:** ✅ Aceptado (Baseline API Architecture - Congelado)

---

## 1. La Frontera del Sistema (API Gateway)
La API de LexIA no está expuesta directamente a Internet. Existe una frontera estricta:
`Internet ➔ WAF ➔ API Gateway ➔ Authentication ➔ Rate Limiting ➔ API ➔ Bus`
El negocio no lidia con certificados, tokens malformados o ataques DDoS. La API asume tráfico limpio e identificado.

---

## 2. Abstracción de Interfaces (The Four Surfaces)
Se definen cuatro superficies independientes, agnósticas de la tecnología que las implemente:
- **Command Interface:** Expone la capacidad de alterar el sistema (REST en el MVP).
- **Query Interface:** Expone lectura rica sobre proyecciones (GraphQL en el MVP).
- **Subscription Interface:** Notifica al cliente de cambios asíncronos (SSE / WebSockets).
- **Integration Interface:** Capa aislada para interoperabilidad (SPOA, INPEC).

---

## 3. Aislamiento DTO vs. Evento
El contrato HTTP de la API **jamás** debe mapear uno a uno al *Domain Event*.
`HTTP DTO ➔ API Mapper ➔ Command ➔ Domain ➔ Event`
Aislar el DTO HTTP del esquema interno garantiza que los consumidores de internet no se acoplen a la estructura de nuestra base de datos.

---

## 4. Control de Acceso Tripartito
La seguridad opera en tres compuertas secuenciales:
1. **Authentication:** ¿Quién eres? (Gateway: Valida JWT).
2. **Authorization:** ¿Tienes el rol técnico/RBAC para llamar este endpoint? (API: Rol "JUEZ_CONOCIMIENTO").
3. **Capability Evaluation:** Aunque tengas el permiso técnico, ¿El dominio lo permite en este instante? (Dominio: `PuedeEmitirSentencia = DENIED` por apelación).

---

## 5. Arquitectura de Commands (Mutaciones)
Todo comando que altere el sistema fluye por esta capa.
- **Asincronía Total:** Retorna siempre `202 Accepted`.
- **Long Running Operations:** La respuesta incluye un `OperationId`. El cliente consulta `GET /operations/{id}` para obtener estados: `Pending`, `Running`, `Completed`, `Rejected`, `Failed`.
- **Bulk Commands:** Operaciones masivas (ej. notificar 400 procesos) se exponen como comandos de lote asíncronos, evitando saturar conexiones HTTP.
- **Trazabilidad:** Cada Request exige o autogenera `X-Correlation-Id`, `X-Causation-Id`, `X-Request-Id`, `X-Tenant-Id`, `X-User-Id` e `Idempotency-Key`.
- **Auditoría Estricta:** Todo Request HTTP genera un registro de *Audit Log* técnico independiente de si el comando fue aceptado o rechazado por el dominio.

---

## 6. Arquitectura de Queries (Lecturas)
Toda consulta obedece a las siguientes reglas:
- **Cursor Pagination:** Prohibida la paginación por *Offset*. Todo listado masivo usa cursores opacos.
- **Consistency Hints:** Dado que leemos de proyecciones asíncronas, la API retorna Headers informativos para que la UI sepa qué tan frescos son los datos:
  - `X-Projection-Version`
  - `X-Event-Checkpoint`
  - `X-Projection-Lag-Ms`

---

## 7. Error Model y Explicabilidad
Basado en el motor de *Capabilities* y el *Temporal Engine*:
```json
{
  "code": "CapabilityDenied",
  "reason": "PendingAppeal_SuspensiveEffect",
  "blockedByEventId": "evt-7890",
  "legalRuleId": "rule-cpp-177",
  "correlationId": "req-1234",
  "explanation": ["Existe un recurso con efecto suspensivo concedido."]
}
```

---

## 8. Gobernanza y Ciclo de Vida de APIs
Las superficies de API obedecen su propio ciclo evolutivo, separado de los eventos:
`Experimental ➔ Accepted ➔ Deprecated (incluye header Sunset) ➔ Removed`.
Ningún endpoint puede leer un Aggregate. Ningún endpoint puede modificar un Read Model de forma directa. La especificación (OpenAPI/GraphQL Schema) es la única fuente de verdad generada por código.
