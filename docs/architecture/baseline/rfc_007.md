# RFC-007 — Universal Domain & Event Sourcing (Arquitectura Táctica V2.0)

**Metadatos de Ingeniería Organizacional:**
- **Autor Principal:** Staff Backend Engineer
- **Revisores:** Chief Architect (CTO), AI Lead
- **Aprobadores:** CTO
- **Estado:** ✅ Aceptado (Baseline Architecture - Congelado)

---

## 1. El Flujo Arquitectónico (CQRS Explícito)
La plataforma obedece estrictamente al patrón Command Query Responsibility Segregation (CQRS):
`Commands (Intención) ➔ Domain (Valida Reglas) ➔ Events (Hechos) ➔ Projections (Vistas) ➔ Queries (Lecturas UI)`

---

## 2. Bounded Contexts y Anti-Corruption Layer (ACL)
1. **Process Management Context:** Ciclo de vida del Proceso y Etapas.
2. **Time & Scheduling Context:** Gestión matemática de Términos y Audiencias.
3. **Decision & Content Context:** El núcleo de la lógica jurídica y su materialización.
4. **Identity & Access Context:** Roles y Actores.
5. **AI & Knowledge Context:** Motor de Coherencia, RAG.
6. **Integration Context (Anti-Corruption Layer - ACL):** Ningún modelo externo (SPOA, INPEC, TYBA) entra al dominio. El ACL traduce modelos externos a `Commands` o `Events` internos del Lenguaje Ubicuo de LexIA.

---

## 3. Aggregates (Cohesión y Reducción de Concurrencia)
Se rechazan los agregados monolíticos. Diferentes raíces de agregado comparten el mismo `ProcessId`.
1. **Aggregate: `Process`**: Identidad del caso, Jurisdicción, Etapa, Actores.
2. **Aggregate: `Decision`**: Razonamiento jurídico independiente del formato.
3. **Aggregate: `Document`**: Representación material (PDF, Word), Hash, Firmas.
4. **Aggregate: `Term`**: Fechas, duración, suspensiones, expiración.
5. **Aggregate: `Hearing`**: Agenda, desarrollo de la sesión, incidencias.

---

## 4. Domain Services
Lógica de negocio que no pertenece naturalmente a ninguna entidad única:
- `DeadlineCalculator`, `JurisdictionResolver`, `RiskEvaluator`, `DocumentAssembler`.

---

## 5. Commands y Eventos (Estructura Meta)
**Invariante Arquitectónica:** *El Event Store almacena hechos de negocio. Es diferente del Audit Log (que almacena IPs, navegadores y clicks del usuario).*
- **Eventos Clave:** `ProcessCreated`, `HearingScheduled`, `DecisionApproved`, `DecisionSigned`.

---

## 6. Policies (Desacoplamiento Absoluto)
Las Políticas reaccionan a Eventos de Dominio mediante un Event Bus (Pub/Sub). No se conocen entre sí.
`Evento ➔ Bus ➔ Policy A / Policy B / Policy C`

---

## 7. Invariants y Restricciones Arquitectónicas
1. **La IA NO escribe en el Event Store:** Ningún modelo de IA tiene permiso de base de datos. Toda sugerencia es un `Command`.
2. **Independencia de Decisiones:** Una sentencia no puede firmarse sin un evento `DecisionApproved`.
3. **Restricción de Términos:** No se puede iniciar un término sin un evento causal explícitamente vinculado.

---

## 8. Read Models / Projections y Snapshot Strategy
- **Regla de Oro:** Toda proyección debe ser efímera. Debe poder ejecutarse `DELETE Projection` y reconstruirse al 100% procesando nuevamente el Event Store desde cero (`Replay`).
- **Snapshot Strategy (Performance):** Snapshot en caché cada **cambio macro de Etapa** o cada **200 eventos**.

---

## 9. Versionado del Dominio
- **Versionado de Eventos:** Upcasting para leer eventos viejos.
- **Versionado de Dominio (Domain V1 ➔ Domain V2):** Procesos históricos persisten bajo la ley antigua (Domain V1); procesos nuevos bajo la ley nueva (Domain V2).

---

## 10. ANEXOS TÉCNICOS (Hard Architectural Rules)

### Anexo A — Transaction Boundary
Cada **Command** se ejecuta dentro de **una única transacción de un Aggregate**.
`Carga Aggregate ➔ Valida Invariantes ➔ Genera Eventos ➔ Append al Event Store ➔ Commit.`
Nunca se actualiza SQL, Redis o Elastic de forma síncrona en esta transacción; todo lo demás es asíncrono.

### Anexo B — Idempotencia
Toda la plataforma asume reintentos y fallos de red. `SignDecision` invocado dos veces por un timeout de UI debe producir exactamente un único evento `DecisionSigned`. No genera duplicados.

### Anexo C — Optimistic Concurrency
Cada append al Event Store requiere un `ExpectedVersion`.
`ExpectedVersion = 41 ➔ append() ➔ si actual es 42 ➔ ConcurrencyException.` 
Protege el dominio sin usar bloqueos pesados (locks).

### Anexo D — Event Envelope
Todo evento usará este sobre (envelope) canónico:
- `EventId`, `EventType`, `AggregateId`, `AggregateType`, `Version`, `OccurredAt`, `CorrelationId`, `CausationId`, `UserId`, `TenantId`, `Payload`.

### Anexo E — Dominio Puro
**Regla Estricta:** El dominio NO conoce infraestructuras. El dominio **ignora** la existencia de PostgreSQL, Redis, Kafka, Elasticsearch, OpenAI, Ollama, HTTP, REST o GraphQL. Solo conoce *Commands, Events, Policies, Aggregates, Value Objects y Domain Services*.

### Anexo F — Event Taxonomy (Convención de Nombres)
- **Commands:** Imperativo (`CreateProcess`, `ApproveDecision`).
- **Eventos:** Pasado (`ProcessCreated`, `DecisionApproved`).
- **Policies:** Sufijo Policy (`DecisionSignedPolicy`).
- **Handlers:** Sufijo Handler (`ApproveDecisionHandler`).
- **Projectors:** Sufijo Projector (`TimelineProjector`).
- **Read Models:** Sufijo Projection (`TimelineProjection`).
- **Queries:** Sufijo Query (`GetTimelineQuery`).

### Anexo G — Abstracción del Tiempo (Time)
En la plataforma judicial el tiempo es parte matemática del dominio. Prohibido usar `DateTime.Now` o similar. Todo depende de una interfaz (ej. `IClock` o genérico equivalente) que permite reproducir historia, mockear pruebas y simular cambios.

### Anexo H — Event Store como Única Fuente de Verdad
**Ninguna proyección constituye la verdad del sistema. La única fuente de verdad es el Event Store.** Está terminantemente prohibido hacer operaciones de escritura directa que modifiquen el estado de un Read Model / Projection eludiendo el flujo CQRS.
