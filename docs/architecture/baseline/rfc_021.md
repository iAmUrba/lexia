# RFC-021 — Engineering Handbook & Coding Standards

**Metadatos de Ingeniería Organizacional:**
- **Autor Principal:** Chief Architect (CTO) / Principal Engineer
- **Revisores:** Lead Engineers
- **Estado:** 🔄 Living Document (Evolutivo, no congelado)

---

## 1. Propósito
A diferencia de los RFC-001 al RFC-020 (Constitución congelada), este documento es un manual vivo que evoluciona con el lenguaje y la experiencia del equipo. Describe **CÓMO** se programa LexIA en C# (.NET).

---

## 2. Pureza del Dominio (Cero Dependencias Externas)
El proyecto `LexIA.Domain` es sagrado. No puede tener dependencias externas (NuGet) que contaminen el negocio.
- **Permitido:** `System.*`, `LexIA.Abstractions`.
- **Prohibido:** ORMs (EF Core), Logging (Microsoft.Extensions.Logging), Validation frameworks (FluentValidation), Drivers (Npgsql), o bibliotecas de Inyección de Dependencias.

---

## 3. El Compilador como Primer Revisor (Type Safety)
Toda regla de negocio que pueda expresarse mediante tipos, debe expresarse mediante tipos; nunca mediante comentarios o convenciones tácitas.
- **Strongly Typed IDs:** Usar `readonly record struct ProcessId(Guid Value);` en lugar de `Guid processId;`.
- **Inmutabilidad Absoluta:** Usar `sealed record RegisterHearingCommand(...)` en lugar de clases planas.
- **Tipos estrictos para prevención de errores:** Usar `Nullable Reference Types` (`<Nullable>enable</Nullable>`) y evitar el estado mutable oculto.

---

## 4. Modelado de Dominio en C#
El dominio jurídico es determinista e inmutable.
- **Domain Events:** Todo evento se define como un `record` inmutable. Los nombres deben estar en pasado (*Event Naming: `DecisionSigned`, nunca `SignDecisionEvent`*).
- **Aggregates:** Todo estado interno debe ser de solo lectura para el exterior. La única forma de mutar el estado es a través de métodos de comando que validan invariantes y generan un `Append` de un evento.

---

## 5. Mediación y Comandos (Custom Bus)
- **Prohibido MediatR:** LexIA utiliza un Command Bus propietario. Esto garantiza la inyección obligatoria del `TenantContext` y observabilidad (OpenTelemetry) sin depender del pipeline oscuro de librerías de terceros.
- **FluentValidation:** Se aplica en la capa de Aplicación/Bus ANTES de tocar el Agregado. Si un comando es lógicamente inválido, se rechaza sin instanciar dominio.

---

## 6. Persistencia y Serialización
- **JSONB Serialization:** Los eventos se guardan como cadenas JSON inmutables. El motor `System.Text.Json` se configura para Forward-Compatibility (Ignorar campos extra) y rechazar ausencia de propiedades obligatorias.
- **Event Versioning:** Los eventos emitidos no se modifican jamás. Si el esquema evoluciona con el tiempo, se crea una nueva versión en código (`HearingRegisteredV2`).

---

## 7. Pruebas y Fitness Functions
- **Unit & Replay Tests (`xUnit`):** Todo Agregado debe contar con *Replay Tests* (Reconstrucción exacta de estado desde la historia).
- **Architecture Tests (`NetArchTest`):** Fitness Functions obligatorias en CI. Ejemplo: El dominio falla al compilar si referencia infraestructura.
