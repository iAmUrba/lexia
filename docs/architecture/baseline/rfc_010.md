# RFC-010 — Event Platform & Persistence Architecture (V2.0)

**Metadatos de Ingeniería Organizacional:**
- **Autor Principal:** Staff Backend Engineer
- **Revisores:** CTO, DBA, Cloud Architect
- **Aprobadores:** Chief Architect (CTO)
- **Estado:** ✅ Aceptado (Baseline Infrastructure - Congelado)

---

## 1. Principios de Persistencia (Hard Rules)
- **Append-Only:** Los eventos solo se insertan al final de un stream.
- **Inmutabilidad Absoluta:** Un evento persistido NUNCA se actualiza (`UPDATE` prohibido).
- **Eliminación Lógica:** Toda eliminación es mediante tombstones (nunca `DELETE`).
- **Determinismo de Replay:** Reproducir un stream debe generar siempre el mismo Aggregate.
- **Cero Accesos Directos:** Prohibidas las lecturas directas a la tabla subyacente. Todo pasa por `IEventStore`.

---

## 2. El Pipeline de Persistencia y Cadena de Integridad

### A. Separación de Modelos
El modelo de dominio nunca se guarda directamente. Atraviesa tres niveles de abstracción:
`Domain Event ➔ Event Envelope ➔ Storage Record`
El `Storage Record` es la representación binaria adaptada a la base de datos (PostgreSQL), permitiendo migrar a otra DB sin afectar al `Event Envelope`.

### B. Serialización Desacoplada
`Envelope ➔ Serializer ➔ Binary/JSONB ➔ Persistence`
Esta capa permite inyectar compresión, cifrado o cambiar entre JSON, Protobuf o Avro dinámicamente.

### C. Cadena de Integridad (Evidential Hash Chain)
Dada la naturaleza judicial, los eventos forman una cadena probatoria infalsificable:
`E1 (Hash) ➔ E2 (Hash = SHA256(E2_Data + E1_Hash)) ➔ E3...`
Si un usuario con acceso root altera un registro en PostgreSQL, toda la cadena deja de validar matemáticamente.

---

## 3. Identidad y Metadatos del Stream
Cada Agregado tiene su propia línea temporal (ej. `Process-00145`, `Term-qwe-12`).
Para evitar leer eventos solo por telemetría, cada Stream posee **Stream Metadata**:
- `StreamId`, `AggregateType`, `CreatedAt`, `LastVersion`, `SnapshotVersion`, `LegalRegimeId`, `RetentionPolicy`, `PartitionKey`.

---

## 4. Arquitectura MVP y Persistencia (PostgreSQL)
**Decisión Técnica:** El MVP utilizará **PostgreSQL (Append-only + JSONB)** oculto tras `IEventStore`.
- **Control de Concurrencia:** Optimistic Concurrency nativo mediante un constraint único de `StreamId + Version`.
- **Estrategia de Escalabilidad (Conservadora):** El Sharding físico multicúmulo queda pospuesto. El MVP escalará mediante una única gran base de datos con particionado interno nativo e índices optimizados, complementada con réplicas de lectura. Cuando la evidencia demuestre saturación, se abrirá un ADR para sharding por Distrito Judicial.

---

## 5. Event Store Interface (IEventStore)
```csharp
interface IEventStore {
    Task AppendAsync(string streamId, IEnumerable<EventEnvelope> events, long expectedVersion);
    Task<IEnumerable<EventEnvelope>> ReadStreamAsync(string streamId, long fromVersion = 0);
    Task SubscribeToAllAsync(long fromCheckpoint, Func<EventEnvelope, Task> handler);
    Task SaveSnapshotAsync(string streamId, Snapshot snapshot);
    Task<Snapshot> GetLatestSnapshotAsync(string streamId);
}
```

---

## 6. Snapshot Architecture
No es solo un archivo de estado. Los snapshots poseen Metadata estricta para garantizar compatibilidad:
- `SnapshotId`, `AggregateVersion`, `AggregateSchemaVersion`, `CreatedAt`, `Hash`, `EventRange`.
Si el código detecta un `AggregateSchemaVersion` incompatible con la versión actual del código, el Snapshot se invalida y el Aggregate se reconstruye desde cero de forma transparente.

---

## 7. Projection Platform y Versionado
Las proyecciones son las vistas (Timelines, Workspaces) que lee la UI.
**Versionado de Proyecciones:**
Nunca se sobrescribe una proyección en producción si hay cambios estructurales. Existen simultáneamente:
`TimelineProjection V1` y `TimelineProjection V2`.
El Projection Worker construye la V2 silenciosamente desde el Event Store mientras la UI sigue leyendo la V1. Una vez la V2 alcanza el *head* del stream, se cambia el enrutador y se elimina la V1 (Zero-Downtime Replay).

---

## 8. Modos Operativos y Replay
El sistema debe comportarse de forma predecible según el estado operativo:
- **Normal:** Lectura y escritura activas.
- **Maintenance / Read-Only:** Commands rechazados; Queries permitidos.
- **Recovery:** Commands rechazados; Projections reconstruyéndose.

**Replay Modes (Modos de Reproducción):**
- **Full Replay:** Desde el Evento 0.
- **Aggregate Replay:** Solo para un `StreamId` específico.
- **Projection Replay:** Solo alimenta una vista particular.
- **Timeboxed Replay:** Desde `Fecha A` hasta `Fecha B` (Crítico para auditorías jurídicas).

---

## 9. Archival Strategy (Ciclo de Datos)
Procesos de más de 15 años no deben penalizar el almacenamiento SSD NVMe.
- **Hot:** Event Store primario (Procesos Activos).
- **Warm:** Storage secundario (Procesos Archivados Recientes).
- **Cold / Archive:** Blob Storage (S3). Los replays siguen funcionando, pero asumen latencia de red mayor.

---

## 10. Gobernanza y Observabilidad
- **Métricas:** Append Latency, Projection Lag, Optimistic Concurrency Failures.
- **Gobernanza:** *Ninguna modificación al almacenamiento físico, serialización o recuperación puede implementarse directamente. Requiere un ADR aprobado por Arquitectura.*
