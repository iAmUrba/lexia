# RFC-008.5 — Temporal Reasoning Engine

**Metadatos de Ingeniería Organizacional:**
- **Autor Principal:** Staff Backend Engineer / Legal Engineer
- **Revisores:** Domain Expert Council
- **Aprobadores:** Chief Architect (CTO)
- **Estado:** ✅ Aceptado (Baseline Temporal Engine - Congelado)

---

## 1. La Capa de "Temporal Facts"
Para asegurar un desacoplamiento absoluto, el motor temporal jamás lee eventos de negocio directamente. Existe un pipeline de traducción:
`Domain Events ➔ Legal Facts ➔ Temporal Facts ➔ Temporal Engine`
- *Domain Event:* `DecisionSigned`
- *Legal Fact:* `DecisionBecomesNotifiable`
- *Temporal Fact:* `DeadlineStarted (TermId, TemporalRuleId)`
Esto aísla la matemática del significado jurídico.

---

## 2. Abstracción Absoluta (Agnosticismo Legal)
**Regla de Oro:** El Motor Temporal ignora por completo la existencia de la Ley 906, Ley 600 o cualquier código. 
Solo conoce una abstracción matemática llamada `TemporalRule`:
```json
{
  "ruleId": "rule-cpp-177",
  "priority": 1,
  "parameters": { "duration": 5, "type": "working_days" }
}
```
Si el Congreso emite una nueva ley, se inyecta un nuevo `TemporalRule` asociado a un `LegalRegime`, sin alterar una sola línea de código del Motor.

---

## 3. Composición de Estrategias y el "Institutional Timeline"
El cálculo no es un bloque monolítico. Es un pipeline de **Estrategias Sustituibles**:
`Calendar Strategy ➔ Counting Strategy ➔ Suspension Strategy ➔ Expiration Strategy`.

**Incertidumbre Institucional:** 
Más allá de las suspensiones de un expediente único, LexIA incorpora el `InstitutionalTimeline`. 
Eventos masivos (paro judicial, hackeo nacional, pandemia) se registran en esta línea temporal transversal. El motor cruza el expediente contra el `InstitutionalTimeline` y recalcula automáticamente los términos de *miles* de procesos sin intervención manual.

---

## 4. Calendarios Versionados
No existe "el calendario" estático. Existe el `CalendarVersion`.
Los festivos y acuerdos de vacancia cambian cada año. Cada proyección temporal usa el `WorkingCalendar` con la versión exacta que estaba vigente en el momento de los hechos.

---

## 5. El Tiempo Produce Eventos (Domain Events Reactivos)
El paso del tiempo no solo alimenta vistas (Read Models). El motor cronológico emite eventos de dominio activos:
- `DeadlineExpired`
- `DeadlinePaused`
- `DeadlineResumed`
- `DeadlineWarningGenerated`
- `RiskThresholdReached`
Otras *Policies* del sistema reaccionan a estos eventos para ejecutar acciones (Ej. Notificar al Juez o generar un auto de incumplimiento).

---

## 6. Explicabilidad (XAI) y Trazabilidad Matemática
El tiempo en LexIA debe ser una explicación demostrable, no solo un resultado.
La API de cálculo no retorna una fecha cruda. Retorna un objeto rico `TemporalProjection`:
```json
{
  "calculationId": "calc-8849-xyz",
  "deadline": "2026-10-15T23:59:59",
  "elapsedDays": 3,
  "remainingDays": 2,
  "calendarVersion": "2026-v2",
  "explanation": [
    "Inició el 3 de mayo (Día Cero aplicado).",
    "Feriado nacional omitido el 5 de mayo.",
    "Suspensión institucional (Paro) del 8 al 12.",
    "Reanudado el 13 de mayo."
  ]
}
```
El `calculationId` es el comprobante matemático (Snapshot) que garantiza que dentro de 10 años, el juez pueda entender exactamente por qué el sistema arrojó esa fecha.

---

## 7. Simulación de Escenarios ("What-if Analysis")
El motor posee la capacidad de operar sobre ramas temporales efímeras (Sandboxes) para responder a hipótesis sin afectar el Event Store oficial.
- *"¿Qué pasa si notifico mañana?"*
- *"¿Cuándo vencería el término si hubiera un paro de 2 semanas?"*
Esta funcionalidad es el eje operativo que utilizará la Inteligencia Artificial (Copiloto) y los Jueces para planeación del despacho.
