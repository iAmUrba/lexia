# RFC-009 — Event Catalog (Contratos Semánticos del Dominio)

**Metadatos de Ingeniería Organizacional:**
- **Autor Principal:** Staff Backend Engineer
- **Revisores:** CTO, QA & Reliability Lead, AI Lead
- **Aprobadores:** Chief Architect (CTO)
- **Estado:** ✅ Aceptado (V1.0 - Baseline Catalog)

---

## 1. Gobernanza del Catálogo (Reglas Fundamentales)

**La Regla de Oro Arquitectónica:**
> *"Ningún RFC posterior (010, 011...) puede redefinir el dominio. Solo puede expresarlo."*
Si al diseñar un payload descubrimos una fricción conceptual, está estrictamente prohibido "parchar" el JSON. Se debe abrir un ADR proponiendo modificar el RFC-008.

**Eventos como Artefactos Versionados:**
De aquí en adelante, cada evento deja de ser texto en un Markdown para convertirse en un archivo ejecutable e independiente (Ej. `events/decision/DecisionSigned.yaml`). El esquema dictará SDKs, contratos de API, auditoría y reglas de CI/CD.

---

## 2. Anexos de Diseño y Estilo

### Anexo A — Event Lifecycle
1. **Draft** ➔ **Experimental** ➔ **Accepted** (Garantiza retrocompatibilidad) ➔ **Deprecated** (Prohibida su emisión, pero soportado) ➔ **Removed**.

### Anexo B — Taxonomía Canónica
Clasificación para navegabilidad: `Core Process`, `Decision`, `Hearing`, `Appeal`, `Deadline`, `Competence`, `Notification`, `AI Advisory`, `Integration`, `Institutional`.

### Anexo C — Event Style Guide
- **Pasado:** Los eventos siempre se nombran en pasado (`DecisionSigned`).
- **Positivo:** Nombres afirmativos (`DecisionSigned`, no `DecisionNotUnsigned`).
- **Singularidad:** Un evento representa un único hecho consumado.
- **Sin Redundancias:** El nombre expresa el hecho. El payload solo aporta contexto. (Prohibido mandar `"status": "SIGNED"` dentro de `DecisionSigned`).
- **Agnóstico Tecnológicamente:** Prohibido usar prefijos de bases de datos o números de versión en el nombre principal del evento.

### Anexo D — Event Review Checklist
Antes de aceptar un archivo `.yaml` de evento nuevo, el PR debe responder `Sí` a:
- [ ] ¿Representa un hecho consumado?
- [ ] ¿Tiene un único significado?
- [ ] ¿Puede reproducirse mediante replay puro?
- [ ] ¿Genera un Legal Fact o Temporal Fact claramente definido?
- [ ] ¿Afecta explícitamente Capabilities?
- [ ] ¿Su payload evita redundancias y obvia estados implícitos?

## 3. Arquitectura del Contrato de Evento
La ficha maestra del evento impone una separación estricta:
1. **Definition:** La Semántica (Por qué existe y qué garantías da).
2. **Schema:** El Payload Técnico (Identificadores de negocio `ProcessId`, `DecisionId` puramente aislados de la Identidad Técnica `EventId`).

### Ficha Canónica y Ejemplos

#### 1. `DecisionSigned`

**DEFINITION**
- **Category:** Decision Events
- **Purpose (Why this exists):** Representar el instante exacto en que una decisión adquiere existencia jurídica firmada y deja permanentemente de ser editable.
- **Semantic Stability:** ★★★★★ Permanent
- **Preconditions:**
  - Existe un evento previo `DecisionApproved`.
  - La Decisión no tiene previamente un evento `DecisionSigned`.
- **Guarantees:**
  - La Decisión jamás volverá a ser editable.
  - El documento adquiere su hash criptográfico final inmutable.
- **Capabilities Afectadas:**
  - Habilita: `PuedeNotificar`, `PuedeApelar`.
  - Inhabilita: `PuedeEditarDecision`.
- **Legal Facts Generados:** `DecisionBecomesNotifiable`, `DecisionHasLegalEffect`.
- **Ordering Requirements:** Per Aggregate (`ProcessId`).
- **Consistency Model:** Strong Ordering para la UI del Workspace; Eventually Consistent para el Notification Service.
- **Privacy Classification:** PII (Personally Identifiable Information).

**SCHEMA / ENVELOPE**
- **Idempotency:** Basado en Hash del documento y `DecisionId`. Un reintento no duplica firmas.
- **Replay Notes:** Crítico para reconstruir el inicio histórico de la ejecutoria.
```json
{
  "eventId": "uuid-tech",
  "processId": "uuid-biz",
  "decisionId": "uuid-biz",
  "signatureHash": "SHA256",
  "signedByRole": "JUEZ_CONOCIMIENTO"
}
```

#### 2. `AppealConcededWithSuspensiveEffect`

**DEFINITION**
- **Category:** Appeal Events
- **Purpose (Why this exists):** Registrar que el Juez de origen concedió la apelación y pierde temporalmente su potestad para emitir actos de fondo mientras sube al Superior.
- **Semantic Stability:** ★★★★☆ Stable
- **Preconditions:**
  - Existe un evento previo `AppealInterposed` sin resolver.
- **Guarantees:**
  - Ninguna Capability que permita avanzar hacia la siguiente Etapa del Grafo Principal podrá estar activa hasta que el Superior resuelva.
- **Capabilities Afectadas:**
  - Inhabilita: `PuedeEmitirSentencia`, `PuedeEjecutarFallo`, `PuedeFijarAudienciaSiguiente`.
- **Temporal Facts Generados:** `TermSuspensionTriggered` (Detiene relojes del proceso principal).
- **Ordering Requirements:** Per Aggregate (`ProcessId`).
- **Consistency Model:** Strong Ordering (Requerido inmediatamente por el Temporal Reasoning Engine).
- **Privacy Classification:** CONFIDENTIAL (Información procesal de alto nivel).

**SCHEMA / ENVELOPE**
- **Idempotency:** Basado en `AppealId`.
```json
{
  "eventId": "uuid-tech",
  "processId": "uuid-biz",
  "appealId": "uuid-biz",
  "appealedDecisionId": "uuid-biz",
  "grantedToActorId": "uuid-biz"
}
```

> [!IMPORTANT]  
> **Aprobación del CTO:**
> Con este nivel de detalle, donde las Precondiciones, Garantías y Requerimientos de Ordenamiento están estipulados, el Event Catalog se vuelve un componente ejecutable. Esto permite congelar el RFC-009 y dar paso al inicio técnico de Infraestructura: el **RFC-010 (Event Store & Persistence)**.
