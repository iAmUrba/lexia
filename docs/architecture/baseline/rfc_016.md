# RFC-016 — Deployment & DevOps Architecture (Cadena de Custodia del Software)

**Metadatos de Ingeniería Organizacional:**
- **Autor Principal:** Staff DevOps Engineer / SRE
- **Revisores:** CTO, SecOps Lead, AI Lead
- **Aprobadores:** Chief Architect (CTO)
- **Estado:** ✅ Aceptado (Baseline DevOps Architecture - Congelado)

---

## 1. Principio Constitucional del Despliegue
> *"El despliegue no distribuye software; distribuye garantías verificadas."*

El despliegue no es un proceso técnico aislado, sino un mecanismo para preservar las garantías institucionales de LexIA.

---

## 2. Environment Promotion (Cero Drift)
Los entornos no se configuran; se promueven. Desarrollo, Integración, Staging y Producción derivan del mismo conjunto de manifiestos y políticas. Toda configuración que exista en Producción y no en Git constituye deuda técnica prohibida.

---

## 3. Configuración como Artefacto Gobernado
El GitOps no se limita a Kubernetes. Toda configuración es un artefacto versionado, con propietario y aprobación obligatoria.
- Se prohíbe la edición "en caliente" en Producción.
- Abarca: *Feature Flags*, SLOs, *AI Policies*, *Prompt Bundles*, *Retention Policies* y *Data Classification Rules*.

---

## 4. Promotion Pipeline y RFC Compliance Verification
Toda versión fluye mediante automatización rígida.
`Source ➔ Build ➔ Static Verification ➔ RFC Compliance ➔ Security ➔ Integration ➔ Performance ➔ AI Evaluation ➔ RC ➔ Staging ➔ Production`

**RFC Compliance Verification (Puertas de Arquitectura):**
El pipeline certifica que la versión cumple la constitución arquitectónica:
- `RFC-008`: ✔ *Capabilities* respetadas (cero lógica dura).
- `RFC-009 / RFC-011`: ✔ *Contract Verification* (Compatibilidad OpenAPI, GraphQL y Eventos validada mediante Contract Testing).
- `RFC-010`: ✔ *Event Store Replay* exitoso en memoria.
- `RFC-012`: ✔ *AI Evaluation* (Groundedness y Hallucination) aprobada.
- `RFC-013`: ✔ *Security Gates* (SLSA, SBOM, Firmas) aprobados.
- `RFC-014`: ✔ *IAM Policies* válidas.
- `RFC-015`: ✔ *Observability Instrumentada* y SLOs cumplidos en Staging.

---

## 5. Inmutabilidad, Compatibilidad y Release Evidence
Toda versión desplegada genera un **Release Evidence** (expediente de despliegue auditable) que contiene:
- El hash del SBOM, la cadena de aprobación y reportes de seguridad/IA.
- **Platform Compatibility Matrix:** Declara compatibilidad cruzada exacta con versiones de: Event Store, Proyecciones, APIs, Model Registry, Prompts y RFC Baseline.

---

## 6. Evolución de Bases de Datos y Zero-Downtime
**Database & Schema Evolution:**
Toda migración de proyecciones o bases operativas es versionada, idempotente y diseñada con compatibilidad hacia atrás/adelante. Las migraciones no pueden bloquear despliegues progresivos.
**Zero-Downtime:** Coexistencia silenciosa de Proyecciones V1 y V2 hasta la sincronización, momento en el cual el Gateway hace switch automático.

---

## 7. Progressive Delivery y Release Readiness
**Entrega Institucional Progresiva:**
`Internal ➔ Pilot Court (Juzgado Piloto) ➔ One District ➔ Several Districts ➔ National`
**Release Readiness Checklist:**
Antes del pase a Producción nacional, el sistema exige comprobación de preparación operacional: ¿Hay runbook? ¿Hay dashboard y alertas? ¿Existe plan de rollback definido? ¿Responsable asignado?

---

## 8. Filosofía de Rollback
La historia es inmutable. Existen 3 niveles, pero se prohíbe tocar el origen de los datos:
1. `Deployment Rollback:` Volver al contenedor/binario anterior.
2. `Feature Rollback:` Apagar el Feature Flag (los cuales tienen clases y fechas de expiración duras).
3. `Projection Rollback:` Rebobinar un *Read Model*.
**PROHIBIDO:** *Event Store Rollback*. Los eventos jamás retroceden.

---

## 9. Chaos Engineering y Resiliencia Continua
No basta con diseñar Disaster Recovery. El pipeline (o tareas programadas) debe inyectar fallos reales periódicamente para demostrar la supervivencia de la plataforma ante:
- Caída de un nodo o pérdida de una réplica de base de datos.
- Rotación de llaves (KMS) y expiración de certificados.
- Reconstrucción completa de proyecciones y restauración *PITR* aislada.

---

## Architecture Decision Record (ADR Summary)
- **Decisión:** Adoptar un modelo de entrega progresiva, artefactos inmutables y GitOps estricto, donde el *Promotion Pipeline* actúa como juez de cumplimiento (*RFC Compliance*) de todos los RFCs de dominio, seguridad, IA y observabilidad.
- **Alternativas descartadas:** Despliegues manuales, pipelines centrados exclusivamente en CI/CD funcional y alteraciones "en caliente" de configuración o infraestructura.
- **Trade-offs:** Incrementa drásticamente el tiempo de compilación y promoción (simulaciones de Replay, evaluación matemática de IA, escaneos SLSA) y requiere mayor madurez del equipo, pero reduce a casi cero la probabilidad de introducir regresiones estructurales.
- **Consecuencias futuras:** Los RFC de operación (RFC-017 Multi-tenancy, RFC-020 Runbooks) podrán asumir de facto que toda versión en producción ya demostró automática y criptográficamente el cumplimiento de la arquitectura diseñada.
- **Métricas de éxito:** 100% de despliegues respaldados por un *Release Evidence*; 0% de configuración editada fuera de Git; Sobrevivencia exitosa a inyecciones periódicas de Chaos Engineering.
